(didn't write this, thought it was interesting, but I disagree) In practice, I've found this not to be true. I've had a few instances where LLMs had issues related to code duplication.

In a recent example, I had a suite of correctness tests and a suite of benchmarks. The benchmarks looked off, but I had correctness tests which I thought would catch things. On a hunch, I guessed that the correctness tests had not been using the same source as the benchmarks. This was correct. The benchmarks were running against a version of the software I was testing with different parameters.

According to the author, this sort of thing should be caught by modern tools. I don't think it is. In fact, I think DRY might become even more important. At some level of experience, you stop doing DRY as much. You learn that some code duplication is okay.

With LLMs, I think it makes sense more often to DRY. The code complexity that comes with accidentally tying semantically different problems together becomes less relevant with a machine that has 32k tokens in its context window. Whereas the iteration loop of it constantly not realizing why something is working (perhaps again because the whole codebase is not in its context window) is only avoided by constantly deduplicating.

I have a workflow where I do an LLM "defrag" quite often to clean up these issue.
